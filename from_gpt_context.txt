6. Now Start with config.py

We’ll make this clean and scalable.

Your app/config.py should:

    load env variables

    store model name

    store timeout/retry settings

    store docs path candidates

    centralize all runtime config

Use this code:

from functools import lru_cache
from typing import List

from pydantic import Field
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    """
    Central application settings for the Sprint 2 healing engine.

    Values are loaded from environment variables and `.env`.
    """

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )

    # App metadata
    APP_NAME: str = "reMorph"
    APP_ENV: str = "development"
    DEBUG: bool = True

    # LLM provider settings
    GROQ_API_KEY: str = ""
    LLM_MODEL: str = "groq/llama3-8b-8192"

    # Network / request behavior
    REQUEST_TIMEOUT_SECONDS: int = 10
    MAX_FETCH_RETRIES: int = 2

    # Candidate documentation paths to probe
    DOC_PATH_CANDIDATES: List[str] = Field(
        default_factory=lambda: [
            "/openapi.json",
            "/swagger.json",
            "/v3/api-docs",
        ]
    )

    # Healing behavior
    ENABLE_AUTH_HEALING: bool = True
    ENABLE_ROUTE_HEALING: bool = True
    ENABLE_PAYLOAD_HEALING: bool = True

    # Logging
    LOG_LEVEL: str = "INFO"


@lru_cache
def get_settings() -> Settings:
    """
    Cached settings accessor so config is loaded once per process.
    """
    return Settings()

If we are building the **Enterprise MVP of LumenInfra**, we do not start with the AI. We start with the plumbing. If the proxy doesn't work, the AI is useless.

Here is your **4-Sprint Engineering Workflow** to build the production-ready MVP. Treat this as your Jira board.

---

### Sprint 1: The Dumb Gateway (The Plumbing)

**Goal:** Build a reverse proxy that sits between a client and a target API, passes traffic seamlessly, and successfully "traps" error codes.

**Tech Stack:** Python + FastAPI + HTTPX.

- **Task 1.1: Setup the Target Sandbox.** Create a dead-simple, separate mock API (e.g., a dummy Stripe API) that you can intentionally break.
- **Task 1.2: Build the Forwarding Proxy.** Write a FastAPI server that takes an incoming request, forwards it to the mock API, and returns the response to the user.
- **Task 1.3: Implement the Interceptor.** Add middleware to your proxy. If the target API returns `200 OK`, pass it back. If it returns `400` or `404`, *do not send the error to the user yet*. Trap it, print "Drift Detected!" to your console, and return a temporary `503 Healing in Progress` status.
- **Definition of Done:** You can send a request through your Gateway to the target API, and your Gateway successfully intercepts errors without crashing.

### Sprint 2: The Reasoning Engine (The Operator)

**Goal:** Connect your trapped errors to an LLM so it can read the docs and generate a fixed payload.

**Tech Stack:** LiteLLM (or LangChain) + Llama-3 (via Groq/OpenAI for rapid MVP testing, or local vLLM).

- **Task 2.1: The Schema Scraper.** Write a function that, upon intercepting an error, automatically fires a `GET` request to the target API's `/openapi.json` or `/docs` endpoint to pull the current schema.
- **Task 2.2: The Prompt Engineer.** Create the rigid prompt: *"You are the Lumen Operator. The user tried to send [Failed Payload]. The API returned [Error]. The current API schema is [Scraped Schema]. Output ONLY a fixed JSON payload."*
- **Task 2.3: The Re-execution Loop.** Take the LLM's fixed JSON payload and have the Gateway automatically fire a *second* request to the target API.
- **Definition of Done:** You send a broken request to your Gateway. It traps it, asks the LLM for a fix, re-sends the fixed payload, gets a `200 OK`, and returns the success to you. *The Self-Healing loop is closed.*

### Sprint 3: The Audit Dashboard (Governance)

**Goal:** Prove to enterprise clients that you aren't secretly breaking their data. Give them visibility.

**Tech Stack:** SQLite/PostgreSQL + React/Next.js (or HTML/Tailwind SPA).

- **Task 3.1: Database Logging.** Every time Sprint 2 happens, log the event to a database: `Original Request`, `Error Caught`, `LLM <think> Trace`, `Fixed Request`, `Final Status`.
- **Task 3.2: Build the UI.** Create a clean dashboard that pulls from this database.
- **Task 3.3: Visual Diffs.** Implement a UI component that highlights the exact changes the LLM made (e.g., showing `customer_id` crossed out in red, and `account_uid` highlighted in green).
- **Definition of Done:** You can open a browser, see a list of "Healed Routes," and click one to read exactly why the AI made the decision it did.

### Sprint 4: The Continuous Flywheel (The Moat)

**Goal:** Move from using a standard API model (like GPT-4) to your own continuously fine-tuned model.

**Tech Stack:** OpenEnv + HuggingFace TRL (GRPO) + Unsloth.

- **Task 4.1: Export Golden Trajectories.** Write a script to export all the successful "Heals" from your Sprint 3 database into a `.jsonl` dataset.
- **Task 4.2: OpenEnv Chaos Generation.** Use the OpenEnv wrapper we discussed earlier to generate synthetic variations of those real-world breaks.
- **Task 4.3: The Nightly Build.** Set up the Colab/RunPod script to run GRPO on your synthetic data, updating your base Llama-3 model's weights.
- **Definition of Done:** You have a script you can run that trains a model to be better at healing based on yesterday's trapped errors.
